This specification defines a compile-time governance framework for organizations deploying LLMs under regulatory frameworks including the EU AI Act [[EU-AI-ACT]], ISO/IEC 42001 [[ISO-42001]], the UK AI regulatory framework [[UK-AI-REG]], and Canada's Directive on Automated Decision-Making [[TBS-ADM]].

The framework models LLM governance as a typed dependency graph where regulatory obligations flow through control objectives, compliance rules, classification policies, and deployment constraints to produce verifiable compliance artifacts. A single graph definition yields 8 W3C standard outputs: SHACL [[shacl]] validation reports, ODRL [[odrl-vocab]] classification policies, PROV-O [[prov-o]] provenance chains, DCAT [[vocab-dcat-3]] deployment catalogs, Verifiable Credentials [[vc-data-model-2.0]] attestations, SKOS [[skos-reference]] governance vocabularies, OWL-Time [[owl-time]] scheduling intervals, and JSON-LD [[json-ld11]] linked data.

This specification builds on the apercue core specification, which defines the underlying data model, graph patterns, and W3C projection mechanisms. The LLM Governance Framework is a domain-specific application of those patterns to the AI compliance domain.

This is an unofficial specification produced by the apercue.ca project. It is generated from CUE source via cue export ./spec/llm-governance/ -e spec_html --out text.

The specification is informed by the regulatory frameworks referenced in the bibliography. Cross-references to specific articles and clauses are provided for traceability; the framework is designed to accommodate regulatory evolution without structural changes.

Introduction

The Regulatory Landscape

Organizations deploying LLMs in 2025–2026 face a converging set of regulatory requirements:

  • The EU AI Act [[EU-AI-ACT]] (fully applicable August 2026 for high-risk systems) requires risk classification, technical documentation, human oversight, and registration with national authorities.
  • ISO/IEC 42001 [[ISO-42001]] provides a certifiable AI management system framework with planning, operation, performance evaluation, and improvement clauses.
  • The UK AI regulatory framework [[UK-AI-REG]] establishes five principles — safety, transparency, fairness, accountability, contestability — enforced by existing sectoral regulators.
  • Canada's Directive on Automated Decision-Making [[TBS-ADM]] requires Algorithmic Impact Assessments and imposes transparency and recourse obligations scaled by impact level (I–IV). The proposed AIDA [[CAN-AIDA]] would extend these requirements to the private sector.
  • Australia's AI Ethics Principles [[AU-AI-ETHICS]] define eight voluntary principles that are expected to become mandatory through sectoral regulation.
  • The NIST AI RMF [[NIST-AI-RMF]] provides a four-function governance model (Govern, Map, Measure, Manage) widely adopted as a baseline in North America.

These frameworks share common structural elements — risk assessment, control objectives, documentation requirements, audit trails, and reporting obligations — but express them in different vocabularies with different clause numbering.

The Compliance Challenge

Current approaches to LLM governance fall into three categories:

  1. Document-driven compliance: Policies written in natural language documents (Word, PDF, wiki). Cannot be validated automatically. Drift from implementation is invisible until audit.
  2. Platform-driven compliance: SaaS governance platforms with dashboards, risk scores, and reporting templates. Compliance state is a metric in a platform rather than a structural property of the system. Difficult to version-control or diff.
  3. Runtime enforcement: Guardrail systems that filter prompts and responses at inference time. Essential for operational safety but reactive — they detect violations rather than preventing them structurally.

None of these approaches provide compile-time governance: a model where regulatory requirements are expressed as structural constraints that must be satisfied before the system can be deployed. A model that is version-controlled, diffable, auditable, and machine-verifiable.

Compile-Time Governance

This specification defines LLM governance as a typed dependency graph. Obligations flow through controls to rules to policies to deployments to audit. The graph is expressed in CUE [[CUE]] and validated at evaluation time. Phase gates prevent advancement until predecessor deliverables are complete. W3C projections produce machine-readable compliance artifacts — not as a reporting layer on top of the system, but as structural properties of the graph itself.

1. Obligations → 2. Impact → 3. Controls → 4. Knowledge → 5. Policies → 6. Deployment → 7. Audit → 8. Reporting

The result: an organization cannot claim compliance without the graph validating. The compliance state is not a dashboard metric — it is a structural invariant enforced by the type system.

Terminology

Obligation
A binding regulatory or policy requirement. Obligations are root nodes in the governance graph — they create the requirements that all downstream resources must satisfy.
Control Objective
A measurable control derived from one or more obligations. Controls specify what must be enforced without prescribing how.
Compliance Rule
An enforceable implementation of a control objective. Rules specify the mechanism — CUE constraints for compile-time enforcement, policy assertions for runtime enforcement.
Classification Policy
An ODRL 2.2 policy governing data handling at a specific classification level. Defines what actions are permitted, prohibited, and obligatory for data flowing through LLM systems.
Phase Gate
A checkpoint in the governance graph that requires all predecessor deliverables to be satisfied before downstream work can proceed. Phase gates enforce sequential discipline in the compliance lifecycle.
Conformance Profile
A jurisdiction- or organization-specific configuration that declares which obligations, resource types, and phase gates apply. Profiles enable the same framework to serve different regulatory contexts.
Knowledge Grounding
The practice of anchoring LLM responses to verified factual assertions drawn from authoritative sources. Every grounded fact cites its source, classification, and verification date.
Deployment Model
A configuration describing how an LLM is deployed with bound policies. Deployment models declare provider, infrastructure type, classification ceiling, and required compliance gates.
Provenance Chain
A PROV-O linked sequence of activities, agents, and entities that traces a compliance claim from its regulatory source through control implementation to audit evidence.
Compile-Time Governance
A governance model where compliance constraints are structural properties of the system definition, enforced at evaluation time (CUE unification), before any runtime behavior occurs.

Obligation OBL-NNN

A legal or policy mandate that creates binding requirements. Maps to statutes, directives, regulations, and binding guidelines.

Constraints: Authority (issuing body), instrument (legal basis), requirements list, optional compliance deadline.

W3C mapping: Source nodes in SHACL validation graphs. Authority maps to prov:wasAttributedTo.

Example: EU AI Act Art. 9 risk management obligation, Canadian Privacy Act s.7 use limitation.

ControlObjective CTL-NNN

A measurable control derived from one or more obligations. Links regulatory requirements to implementable technical or process controls.

Constraints: Obligation reference, control type (technical/process/organizational), verification method.

W3C mapping: Focus nodes in SHACL sh:ValidationResult. Control type maps to SKOS concept classification.

Example: PII blocking in LLM output (technical), human-in-the-loop review (process), bias testing cadence (organizational).

ComplianceRule RULE-NNN

An enforceable implementation of a control objective. Compile-time rules are CUE constraints; runtime rules are policy assertions.

Constraints: Control reference, rule type (compile-time/runtime), enforcement mechanism.

W3C mapping: Maps to sh:sourceConstraintComponent in SHACL validation results.

Example: CUE constraint requiring bilingual output quality score ≥ 0.85, policy requiring human review for AIA Level III+.

ClassificationPolicy POL-NNN

An ODRL 2.2 data handling policy governing information classification levels. Defines permissions, prohibitions, and obligations for data flowing through LLM systems.

Constraints: Classification level, permitted actions, prohibited actions, duty conditions.

W3C mapping: Directly maps to odrl:Policy with odrl:permission, odrl:prohibition, odrl:obligation.

Example: Protected B policy prohibiting storage outside sovereign jurisdiction, Unclassified policy permitting cloud inference.

DeploymentModel DEP-NNN

A deployment configuration for an LLM system with bound policies. Declares the provider, infrastructure, classification ceiling, and gate requirements.

Constraints: Provider, deployment type (cloud/on-premise/hybrid), classification ceiling, required gates.

W3C mapping: Maps to dcat:Distribution in DCAT 3 catalog entries. Access constraints map to odrl:hasPolicy.

Example: Cloud API deployment restricted to Protected A with bilingual gate, on-premise open-weight deployment with no classification ceiling.

PolicyFact FACT-NNN

A verified factual assertion an LLM is permitted to make. Grounds LLM output in authoritative sources to prevent hallucination.

Constraints: Claim text, authoritative source reference, citation, confidence level (verified/derived/interpreted), language.

W3C mapping: Maps to prov:Entity with prov:wasDerivedFrom tracing to authoritative sources.

Example: "The Privacy Act s.7 limits use of personal information to the purpose for which it was collected" — verified, citing SRC-001 §7.

AuthoritativeSource SRC-NNN

A document the governance framework trusts as ground truth. Every policy fact must trace to an authoritative source.

Constraints: Title, URL, format, language, publisher, publication date.

W3C mapping: Maps to prov:Entity with role prov:PrimarySource. Catalog entries as dcat:Dataset.

Example: Treasury Board Directive on Automated Decision-Making (2024), EU AI Act Official Journal publication.

DomainScope SCOPE-NNN

A knowledge boundary defining what an LLM deployment is permitted to answer. Declares permitted and excluded topics with rationale.

Constraints: Domain name, permitted topics list, excluded topics list, rationale.

W3C mapping: Maps to SKOS skos:Concept with skos:scopeNote for boundary documentation.

Example: Procurement domain: permitted topics include vendor evaluation criteria; excluded topics include classified contract values.

AuditRecord AUDIT-NNN

A provenance record linking compliance activities to agents, time periods, and evidence. Enables accountability chains from obligation to attestation.

Constraints: Activity type, agent reference, time period, evidence artifacts.

W3C mapping: Maps to prov:Activity with prov:wasAssociatedWith (agent) and prov:used/prov:generated (artifacts).

Example: Monthly bias audit by AI Ethics Board, quarterly AIA reassessment triggered by model update.

TermDefinition TERM-NNN

An authoritative definition the LLM must use verbatim. Prevents LLM-invented definitions for terms with precise legal or technical meaning.

Constraints: Term, definition text, authoritative source reference, language.

W3C mapping: Maps to SKOS skos:Concept with skos:definition and skos:notation.

Example: "Personal information" as defined in Privacy Act s.3, "high-risk AI system" as defined in EU AI Act Art. 6.

Resource Relationships

Resources are connected by depends_on edges that map to dcterms:requires in JSON-LD. The canonical dependency flow is:

Obligation
  → ControlObjective (depends_on: obligation)
    → ComplianceRule (depends_on: control)
      → ClassificationPolicy (depends_on: rules it enforces)
        → DeploymentModel (depends_on: policies, gates)
          → AuditRecord (depends_on: deployment, controls)
            → VerifiableCredential (depends_on: audit evidence)

Knowledge Grounding (parallel track):
  AuthoritativeSource
    → PolicyFact (depends_on: source)
    → TermDefinition (depends_on: source)
  DomainScope (depends_on: obligations, sources)

This flow is not prescriptive — organizations MAY introduce additional dependency edges to model their specific governance topology. The framework requires only that the graph is a directed acyclic graph (DAG) and that all depends_on references resolve.

Phase Architecture

The governance lifecycle is organized into 8 sequential phases. Each phase produces specific deliverables and W3C projection outputs. Phase gates enforce ordering — Phase N+1 cannot proceed until Phase N's gate is satisfied.

PhaseDescriptionDeliverablesW3C Output
Obligation Graph Map the regulatory landscape. Identify all binding obligations (statutes, directives, standards) and their dependency relationships. Root nodes are primary legislation; dependent nodes are derived instruments. Obligation inventory, dependency DAG, authority mapping JSON-LD graph with dcterms:requires edges
Impact Assessment Classify AI systems by risk level and determine impact tiers. EU AI Act uses four risk categories (unacceptable, high, limited, minimal). Canada's AIA uses four impact levels (I–IV). Map each deployment to its applicable tier. Risk classification matrix, AIA level assignments, high-risk system inventory SHACL sh:ValidationReport for classification completeness
Control Objectives Derive measurable controls from obligations and impact assessments. Each control objective traces to one or more obligations and specifies a verification method. Controls are typed as technical, process, or organizational. Control catalog, obligation-to-control traceability matrix, verification criteria SHACL sh:ValidationReport for control coverage
Knowledge Grounding Establish the factual basis for LLM responses. Identify authoritative sources, extract verified facts, define term definitions, and set domain scope boundaries. Every fact cites its source; every term has a canonical definition. Authoritative source registry, fact inventory, term glossary, domain scopes PROV-O provenance chains (prov:wasDerivedFrom)
Policy Engine Define data classification policies as machine-readable ODRL 2.2 policies. Each policy specifies permissions, prohibitions, and duties for a classification level. Policies bind to deployment models via gate requirements. Classification policies (per level), gate definitions, policy-to-deployment bindings ODRL 2.2 odrl:Policy with permissions/prohibitions
Deployment Constraints Model how LLMs are deployed with bound policies and gate requirements. Each deployment declares its provider, infrastructure type, classification ceiling, and which compliance gates must be satisfied before activation. Deployment inventory, gate bindings, provider classification matrix DCAT 3 dcat:Distribution catalog entries
Audit & Provenance Establish accountability chains from obligation through control to evidence. Define audit activities, assign responsible agents, link to evidence artifacts. Provenance chains enable traceability from any compliance claim back to its regulatory source. Audit plan, agent registry, provenance chain templates, evidence catalog PROV-O prov:Activity chains + EARL earl:Assertion test reports
Compliance Reporting Produce verifiable compliance attestations as W3C Verifiable Credentials wrapping SHACL validation reports. Generate catalog entries for registration with regulatory bodies. Publish governance vocabulary as SKOS concept schemes. Verifiable Credentials, SHACL summary reports, DCAT catalog, SKOS governance vocabulary VC 2.0 credentials, SKOS skos:ConceptScheme

Phase Gate Model

Each phase gate is a CUE constraint that validates completeness. A gate is satisfied when all required resources for that phase are present in the graph and pass structural validation. Gates are implemented using the apercue #Charter pattern:

_charter: charter.#Charter & {
  name: "my-llm-governance"
  gates: {
    "obligation-graph": {
      phase: 1
      requires: {
        "gc-llm-framework": true
        "privacy-act": true
        "ai-act-art9": true
      }
    }
    "impact-assessment": {
      phase: 2
      requires: {
        "aia-level-classification": true
        "risk-matrix": true
      }
    }
    // ... phases 3–8
  }
}

Gap analysis (#GapAnalysis) computes the delta between the charter and the current graph, producing a SHACL sh:ValidationReport where each missing resource is a sh:Violation.

W3C Projection Mappings

Each W3C projection serves a specific governance function. The same graph produces all projections — no separate data entry, no synchronization, no drift between compliance artifacts.

StandardGovernance UseOutput TypePhases
SHACL [[shacl]] Compliance validation. Every control objective becomes a SHACL shape; every violation produces a sh:ValidationResult with severity, focus node, and remediation guidance. sh:ValidationReport 2, 3, 8
ODRL 2.2 [[odrl-vocab]] Data classification enforcement. Each classification level (Unclassified, Protected A/B, Confidential, etc.) becomes an ODRL policy with permissions, prohibitions, and duty conditions. odrl:Policy 5
PROV-O [[prov-o]] Audit provenance. Links compliance activities to responsible agents, evidence artifacts, and authoritative sources. Enables end-to-end traceability from obligation to attestation. prov:Activity, prov:Entity, prov:Agent 4, 7
DCAT 3 [[vocab-dcat-3]] Deployment registry. Each LLM deployment becomes a DCAT dataset with distribution metadata, access constraints, and policy references. dcat:Dataset, dcat:Distribution 6
VC 2.0 [[vc-data-model-2.0]] Compliance attestation. Wraps SHACL validation reports in Verifiable Credentials for tamper-evident, machine-verifiable compliance claims. VerifiableCredential 8
SKOS [[skos-reference]] Governance vocabulary. Domain types, control categories, and classification levels project as SKOS concept schemes with hierarchical and associative relations. skos:ConceptScheme, skos:Concept 8
OWL-Time [[owl-time]] Compliance scheduling. Critical path analysis produces time intervals for each phase gate, enabling project managers to identify schedule risks and resource bottlenecks. time:Interval, time:Duration All
JSON-LD [[json-ld11]] Linked data federation. The shared @context enables governance graphs from different organizations to merge cleanly — same vocabulary, same semantics, different data. @context, @id, @type All

Conformance Profiles

A conformance profile adapts the generic framework to a specific regulatory jurisdiction or organizational context. Profiles declare which obligations apply, which resource types are required, and which phase gates must be satisfied.

Profile Definition

A conformance profile is a CUE value that specifies:

  1. Applicable obligations: Which regulatory instruments bind this organization (EU AI Act, Canadian AIDA, UK principles, etc.).
  2. Required resource types: Which domain types (§3) must be present in the governance graph.
  3. Phase gates: Which phase gates (§4.1) must be satisfied, and what resources each gate requires.
  4. Classification levels: Which data classification levels apply (and therefore which ODRL policies are needed).
  5. Audit cadence: How frequently audit activities must be recorded in the provenance chain.
_profile: {
  name: "eu-ai-act-high-risk"
  jurisdiction: "EU"
  applicable_frameworks: ["EU-AI-ACT", "ISO-42001"]
  required_types: {
    Obligation: true
    ControlObjective: true
    ComplianceRule: true
    DeploymentModel: true
    AuditRecord: true
  }
  classification_levels: ["public", "internal", "confidential"]
  audit_cadence: "quarterly"
  gates: {
    // Phase gate definitions per §4.1
  }
}

Reference Implementation: GC Federal Profile

The GC LLM Governance example implements a conformance profile for the Government of Canada. This profile binds:

  • Obligations: Privacy Act, Official Languages Act, TBS Directive on Automated Decision-Making [[TBS-ADM]], CCCS security guidance.
  • Impact levels: AIA Levels I–IV as defined by the Directive.
  • Classification levels: Unclassified, Protected A, Protected B (with ODRL policies for each).
  • Domain types: All 10 types defined in §3, including PolicyFact and TermDefinition for bilingual knowledge grounding.
  • Phase gates: All 8 phases with 52 resources.

The GC profile demonstrates that a jurisdiction-specific governance charter can produce all 8 W3C projections from a single CUE module with no custom code — only domain-specific resource definitions.

Creating Custom Profiles

Organizations SHOULD create conformance profiles that reflect their specific regulatory obligations. The recommended process:

  1. Identify applicable regulatory frameworks (§1.1).
  2. Map framework requirements to obligation resources (Phase 1).
  3. Determine required classification levels for ODRL policies.
  4. Define phase gates with required resources per phase.
  5. Run cue vet to validate the profile against the domain type constraints.
  6. Run gap analysis to identify remaining work.

Profiles for different jurisdictions can be composed — an organization operating in both the EU and Canada can merge obligations from both the EU AI Act and AIDA profiles, with CUE unification automatically deduplicating shared controls.

EU AI Act Alignment

The EU AI Act [[EU-AI-ACT]] uses a risk-based classification system. High-risk AI systems (Annex III) face the most stringent requirements. The framework's 8 phases map to EU AI Act requirements as follows:

  • Risk classification (Art. 6, Annex III) is addressed in Phase 2 (Impact Assessment). Each LLM deployment receives a risk classification that determines downstream requirements.
  • Quality management (Art. 17) spans Phases 3–6, with control objectives, compliance rules, and deployment constraints forming the quality management system.
  • Technical documentation (Art. 11) is produced as a structural property of the graph — the graph is the documentation.
  • Record-keeping (Art. 12) maps to Phase 7 provenance chains. PROV-O activities with timestamps and agent attributions satisfy the logging requirements.
  • Registration (Art. 49) maps to Phase 8 DCAT catalog entries, which provide the structured metadata needed for EU database registration.
August 2026 deadline: High-risk AI system requirements under the EU AI Act become fully applicable on 2 August 2026. Organizations deploying LLMs classified as high-risk under Annex III MUST have conforming governance systems in place by this date.

Commonwealth Alignment

Commonwealth jurisdictions (UK, Canada, Australia) share a principles-based approach to AI governance that complements the EU's rules-based framework:

  • The UK framework [[UK-AI-REG]] assigns enforcement to existing sectoral regulators (ICO, FCA, Ofcom, CMA) applying five cross-cutting principles. The governance graph's phase gates map to these principles, enabling organizations to demonstrate compliance to multiple regulators from the same data.
  • Canada's TBS Directive [[TBS-ADM]] requires Algorithmic Impact Assessments (AIA) with impact levels I–IV. Phase 2 directly models AIA level assignments; higher impact levels trigger additional control requirements in Phases 3–5.
  • Australia's AI Ethics Principles [[AU-AI-ETHICS]] define eight principles (human/societal wellbeing, fairness, privacy, reliability, transparency, contestability, accountability, human oversight) that map across Phases 1, 3, 5, and 7.

The key advantage of graph-based governance for Commonwealth jurisdictions: principles-based regulation requires organizations to demonstrate how they satisfy principles, not merely check boxes. The provenance chains (Phase 7) and verifiable credentials (Phase 8) provide this evidence structurally.

Approaches to LLM Governance

LLM governance is addressed by multiple categories of tools and frameworks. This section describes five approaches, their strengths and limitations, and how they complement each other. Effective governance typically requires tools from multiple categories.

Runtime Guardrails

Examples: Guardrails AI, NVIDIA NeMo Guardrails, OneShield

Approach: Neural-symbolic systems that filter prompts and responses in real time. Define "RAIL" specifications, initialize guards, wrap LLM calls.

Strengths: Real-time enforcement, prompt injection defense, response filtering, model-agnostic.

Limitations: Reactive (post-deployment), no compile-time guarantees, rules defined per-call rather than per-graph, limited regulatory traceability.

Complementary use: Compile-time governance defines the policy graph; runtime guardrails enforce it at inference time. Guardrails AI's RAIL specs can be generated from ODRL policies.

AI Governance Platforms

Examples: Credo AI, FairNow, Arthur AI, Harmonic Security

Approach: Dashboard-driven governance with risk assessment workflows, bias monitoring, compliance reporting, and EU AI Act / NIST alignment tools.

Strengths: Executive visibility, automated risk scoring, regulatory reporting templates, continuous monitoring.

Limitations: Governance logic lives in the platform, not in the code. Compliance state is a dashboard metric rather than a structural invariant. Difficult to version-control or diff.

Complementary use: Graph-based governance provides the structural model that platforms can visualize. SHACL reports export to any dashboard that consumes W3C data.

LLMOps Observability

Examples: LangSmith, MLflow, Arize AI, Weights & Biases, Helicone

Approach: Trace-level logging of prompts, completions, tool calls, and evaluation metrics. Focus on debugging, cost tracking, and performance optimization.

Strengths: Deep operational visibility, experiment tracking, evaluation frameworks, cost management, production debugging.

Limitations: Observability answers "what happened" — not "what should have happened." No policy model, no regulatory mapping, no compile-time validation.

Complementary use: Observability platforms provide the runtime evidence that PROV-O audit chains reference. Trace IDs become prov:Entity artifacts in the governance graph.

Standards Frameworks

Examples: NIST AI RMF, ISO/IEC 42001, EU AI Act, Canadian AIDA

Approach: Regulatory and voluntary frameworks that define governance functions (Govern, Map, Measure, Manage), management system requirements, and risk classification tiers.

Strengths: Authoritative, comprehensive, internationally recognized. Provide the "what" of AI governance.

Limitations: Frameworks define requirements, not implementations. Organizations must bridge the gap between framework clauses and operational controls. No standard machine-readable format.

Complementary use: The LLM Governance Framework implements these standards as a dependency graph. Each phase maps to specific framework clauses. The regulatory crosswalk (§7) provides the mapping.

Compile-Time Governance (this specification)

Examples: apercue LLM Governance Framework

Approach: Declarative dependency graph where obligations flow through controls to rules to policies to deployments to audit. CUE constraints enforce structural invariants at evaluation time. W3C projections produce machine-readable compliance artifacts.

Strengths: Version-controlled, diffable, compile-time validated. One graph yields 8 W3C standard outputs. Regulatory crosswalk is structural, not narrative. Impossible to deploy without satisfying the graph.

Limitations: Requires CUE expertise. Compile-time model cannot enforce runtime behavior (prompt injection, response quality). Graph complexity grows with regulatory scope.

Complementary use: Compile-time governance is the policy layer; runtime tools (guardrails, observability) are the enforcement and monitoring layers. Together they provide end-to-end coverage.

Summary: Complementary Layers

No single approach covers the full governance lifecycle. The recommended architecture combines:

  1. Compile-time governance (this specification) for structural compliance — the policy graph that must validate before deployment.
  2. Runtime guardrails for operational safety — prompt/response filtering at inference time.
  3. Observability platforms for operational visibility — traces, metrics, and cost tracking that feed back into the provenance chain.
  4. Governance platforms for executive reporting — dashboards that consume W3C projection output (SHACL reports, DCAT catalogs) from the compile-time layer.

The compile-time layer produces machine-readable artifacts (JSON-LD, SHACL, ODRL, PROV-O) that the other layers can consume. This creates a governance data pipeline rather than isolated tools.

Security Considerations

Data Classification

Governance graphs contain resource descriptions that reference regulatory obligations, control implementations, and deployment configurations. While the graph itself does not contain protected data, the structure of the graph reveals organizational compliance posture — which obligations are satisfied, which controls are in place, and which deployments exist.

Organizations SHOULD classify governance graph exports according to their information classification policy. ODRL policies defined in Phase 5 SHOULD apply to the governance data itself, not only to the LLM data it governs.

Supply Chain Integrity

The governance graph references external authoritative sources (URLs in AuthoritativeSource resources). These references SHOULD be validated for integrity:

  • Source URLs SHOULD use HTTPS.
  • Source content SHOULD be archived or checksummed at the time of fact extraction.
  • The date_accessed field SHOULD be populated to enable temporal verification.

Identifier Constraints

All resource identifiers inherit the #SafeID and #SafeLabel constraints from the core specification. Domain type ID formats (e.g., OBL-NNN, CTL-NNN) provide additional structural validation via CUE regex constraints.

Privacy Considerations

Governance graphs for LLM systems raise specific privacy concerns:

  • Policy facts may contain verbatim text from legal instruments. While this text is typically public, the selection of facts reveals organizational focus areas.
  • Domain scopes declare permitted and excluded topics, which may reveal sensitive organizational boundaries (e.g., excluded topics in an HR scope).
  • Audit records reference agents (people or teams) responsible for compliance activities. Agent identifiers SHOULD use role-based references (e.g., "AI Ethics Board") rather than personal identifiers where possible.
  • Deployment models may reveal provider relationships and infrastructure choices. Organizations operating in regulated sectors SHOULD consider whether deployment metadata constitutes commercially sensitive information.

The core specification's privacy guidance on @id values and public export applies to governance graphs. Organizations SHOULD use the public/private deployment split to ensure governance graphs with operational detail are not inadvertently published.
