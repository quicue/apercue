What apercue does

apercue is a CUE module that turns typed dependency graphs into W3C-standard linked data. You define resources and their dependencies. CUE computes everything else: topology, critical paths, compliance checks, gap analysis. Every W3C artifact is a different cue export -e expression on the same graph.

Traditional semantic web

Model data in RDF triples. Load into a triplestore. Write SPARQL queries per view. Run a SHACL processor to validate shapes. Serialize to JSON-LD. Five runtime components. Five failure points. Five things to deploy and monitor.

apercue.ca

Define typed resources in CUE. Run cue export -e <projection>. One binary. Zero servers. 17 W3C projections. CUE's constraint lattice does both the querying (replaces SPARQL) and the validation (replaces SHACL) in a single evaluation pass.

Language

CUE concepts you need

CUE is a constraint-based configuration language. If you know JSON and a typed language, you can read CUE. These six patterns appear everywhere in apercue.

struct-as-set

Instead of arrays, apercue uses structs with true values for set membership. O(1) lookup, clean merging via unification, no duplicates possible.

"@type": {Database: true, Monitored: true} depends_on: {"design": true, "review": true} // check membership: if resource["@type"]["Database"] != _|_ { ... }

comprehensions

CUE's for loops build structs or lists at eval time. These replace SPARQL queries. They're deterministic and fully resolved before any output.

// all resources of type "Database" databases: { for name, r in graph.resources if r["@type"]["Database"] != _|_ { (name): r } }

_|_ (bottom)

CUE's "undefined" value. field != _|_ means "this field exists." Used for existence checks, optional field detection, and error handling. If two constraints conflict, the result is _|_.

// does this resource have ancestors? if r._ancestors[target] != _|_ { "yes, it's a descendant" }

hidden fields ( _ prefix)

Fields starting with _ are private to the package. They don't appear in cue export output. Use them for intermediate computations. Public fields are the API surface.

_tasks: { ... } // private: input data _charter: charter.#Charter // private: charter config gaps: charter.#GapAnalysis // public: exported result

unification ( & )

CUE merges two values by intersecting their constraints. This is how you apply a pattern to data. If the data doesn't satisfy the pattern, evaluation fails.

// apply the #Graph pattern to your input graph: patterns.#Graph & {Input: _tasks} // apply critical path analysis to the graph cpm: patterns.#CriticalPath & { Graph: graph Weights: {for name, t in _tasks {(name): t.time_days}} }

definitions ( # prefix)

Types (schemas) in CUE are called definitions. #Graph, #Charter, #Resource are definitions. They constrain what fields are allowed and what values they can hold. They're never exported directly.

// in patterns/graph.cue: #Graph: { Input: {[string]: vocab.#Resource} // ... computed fields } // in your code: graph: patterns.#Graph & {Input: _tasks}

Infrastructure layer

quicue.ca: infrastructure on apercue

quicue.ca imports apercue's generic graph engine and adds infrastructure-specific types, 35 provider templates, and a static API. Same patterns, different domain. The relationship: apercue is the library, quicue.ca is the application.

Four-layer architecture

Definition
vocab/ + patterns/ — Core schemas (#Resource with identity, network, hosting, dependencies) and graph algorithms (#InfraGraph, #ImpactQuery, #BlastRadius, #SinglePointsOfFailure, #DeploymentPlan).
Template
template/*/ — 35 provider-specific action registries. Each provider maps resource types to executable commands. Proxmox, Docker, VyOS, Vault, GitLab, Caddy, etc. Commands have {host} and {container_id} placeholders filled at compile time.
Value
examples/*/ — 19 concrete infrastructure graphs. The datacenter example has 30 resources with 654 resolved commands. Each example is a complete, working graph with impact queries, SPOF detection, and deployment plans.
Interaction
ou/ — Role-scoped views (ops/dev/readonly) with W3C Hydra JSON-LD API. server/ builds a static API: CUE pre-computes all 654 responses, served from Cloudflare Pages. Zero runtime graph traversal.

Key patterns (beyond apercue)

#ImpactQuery

"What breaks if X goes down?" Computes transitive dependents of any resource. Returns the blast radius as a subgraph.

#BlastRadius

Failure propagation analysis. Given a failed node, which layers are affected? How many resources are unreachable?

#SinglePointsOfFailure

Finds resources where failure cascades to multiple dependents. The nodes you need redundancy on.

#DeploymentPlan

Layer-by-layer startup ordering. Which resources can start in parallel at each depth level? Computed from the topology.

#BindCluster

Provider matching by @type overlap. Selects the right command templates for each resource based on its types. No if-then cascades.

#RollbackPlan

If layer N fails during deployment, what's the undo sequence? Reverse of the deployment plan, scoped to affected layers.

Workflow

Commands to start with

Validate

# validate everything (cross-package)
$ cue vet ./... -c

# validate a single example
$ cue vet ./examples/course-prereqs/

Explore

# human-readable summary
$ cue eval ./examples/course-prereqs/ -e summary

# charter gap analysis (is everything done?)
$ cue eval ./examples/course-prereqs/ -e gaps.complete

# critical path summary
$ cue eval ./examples/course-prereqs/ -e cpm.summary

Export W3C JSON-LD

# SHACL validation report
$ cue export ./examples/course-prereqs/ -e gaps.shacl_report --out json

# OWL-Time schedule
$ cue export ./examples/course-prereqs/ -e cpm.time_report --out json

# critical path sequence
$ cue export ./examples/course-prereqs/ -e cpm.critical_sequence --out json

Build site data

# rebuild all site JSON from CUE
$ bash tools/build-site.sh stage _public

# local preview
$ bash tools/deploy-local.sh serve

W3C evidence (all 17 specs)

# export any spec evidence (from project root)
$ cue export ./w3c/ -e evidence.dcat_catalog --out json
$ cue export ./w3c/ -e evidence.shacl --out json
$ cue export ./w3c/ -e evidence.prov_report --out json
Full picture

How it all fits together

apercue is the library. Everything else imports it and adds domain-specific types. The generic graph engine and W3C projections are shared across all downstream modules.

apercue.ca (generic)

Graph engine, W3C projections, charter planning. 78 pattern definitions. 17 W3C specs. Domain-agnostic. import "apercue.ca/patterns@v0"

quicue.ca (infrastructure)

107 pattern definitions. 35 providers. 40+ resource types. Static API with 654 endpoints. Hydra JSON-LD. import "quicue.ca/patterns@v0"

Knowledge graph

quicue-kg

Knowledge graph framework with reasoning and federation patterns on top of apercue.

Housing

cmhc-retrofit

CMHC housing retrofit graphs. Imports apercue for charter-based tracking of retrofit programs.

Additional modules

orche / boot / cab

Multi-site federation, bootstrap sequencing, and Change Advisory Board reporting. All import quicue.ca.
